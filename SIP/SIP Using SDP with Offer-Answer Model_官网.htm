SIP Using SDP with Offer-Answer Model

时间：2011-07-15 12:50 点击：72次 作者：admin 来源：未知

依据RFC3261-13.2.1所述，SIP应用的Offer/Answer模型是成立在对话环境下的。RFC中还特地对Offer/Answer交互有制约：

1.初始Offer必须在INVITE信息或者第一个可靠的非失利型呼应中,C++的可移植性和跨平台开拓[4]：硬件系统相干。注：当初RFC3261中可靠效应只有2**，接下来将讲到1**（除100外）也可为可靠性效应。

2.如果初始Offer在INVITE音信中，Answer必须展现在一个可靠的非失利型呼应中。也许在1**中就带有Answer（但该Answer必须与之后的2**中的统一），UAC将疏忽统一事务尔后涌现的响应中的Answer。

3.假使初始Offer展现在第一个可靠的非失利型呼应中，Answer一定涌现在对该回应着实认音讯中（ACK）。

4.如果曾经发送或接纳对于第一个Offer的Answer，UAC可以延续发送新的Offer；相反的，如果不曾承认对前一个Offer的Answer，不能发送新的Offer。

5.万一曾经发送或领受对于初始Offer的Answer，UAS阻止在尔后统一个事务的回应音讯中带上新的Offer。

依据RFC3262-5所述，对于Offer/Anwer模型引入了新的伸展。

1.假使INVITE信息带有Offer，UAS一定在一个可靠的非失利型的回应中供给Answer。这将在吼叫完结之前产生好会话。同样，Answer能够展现在1**中，因为RFC3262供给了PRACK方式来保障1**音讯为可靠音信。

2.如果UAC接收到1**中的Offer，必须在PRACK方式中带有Answer。然而如果UAC收到1**中的Answer，则也许在PRACK带上新的Offer。如果UAS接收到PRACK中的Offer，则必须在2**中带上Answer。

3.如果Offer/Answer交互胜利的话，在INVITE事务没有完结之前也能产生好会话。

4.如果在1**中带有Offer的话，UAS在不曾收到对1**着实认之前不能发送2**。

依据RFC3264所述，Offer产生媒体抉择项（高层如SIP供给），而Answer端可以领受或谢绝，取决于高层。UA能够穿过新的Offer/Answer交互来举行会话更新，但旧的Offer/Answer交互未完结之前不可发动新的Offer。

1.发最初始Offer：固然SDP(RFC4566)准许在一个SDP音信中扶持多个会话，但拥有Offer/Answer模型的SDP信息只能包罗一个对话刻画。

2.Offer可以包罗0个或更多的媒体流（每个媒体流运用一个“m=”行和相干的属性来刻画的）。0个媒体流代表只是衔接,“威震敌胆”??炮九团光芒历史连载，尔后能够穿过新的Offer来更新会话。

3.建立单点散布媒体Offer：1）媒体流方向，包罗recvonly、sendrecv（默许）、sendonly及inactive。但必要留神的是在RTP协定中，RTCP不会受该方向波及，即任何安设情形下均处于职业形势。

4.对于recvonly和sendrecv方向的媒体流来说，Offer中的端口号码和地址指示了提供方接纳媒体流的地方。对于sendonly方向的媒体流来说，Offer中的端口号码和地址间接指示了提供方接纳RTCP的地方（除了特异解释,sy.syxinhao.com，RTCP的端口比对应RTP端口高一位）。如果端口是0，则只提供、谢绝或终止媒体流。

5.对于sendonly和sendrecv流来说，Answer也许对于统一编码指示不同的负载类别编号（PayloadType Number），在这种情形下，Offer一定采纳Answer中的编号值。

6.对于RTP流，媒体刻画必要运用"a=rtpmap"来照射RTP媒体负载类别编号，万一不曾对应的"a=rtpmap"行，则应用目前默许的搭配（RFC1890）。

7.“m=”行中所列的编码必须采纳优先次序排行。

8.对于Offer中的每个“m=”行，Answer中务须有对应的“m=”行。Answer中的“t=”行务必与Offer行的统一。

9.谢绝某个Offer的流，该流的端口值必须安设为0.

10.Offer如果是sendonly，则Answer务须为recvonly或许inactive,SSH协定_29933；Offer如果是recvonly，则Answer务必为sendonly或许inactive；Offer如果是sendrecv，则Answer务须为sendrecv、recvonly、sendonly或许inactive；Offer如果是inactive，则Answer务必为inactive。

11.即便Offer是senonly型，Answer的地址和端口也必须存留，因为必要传递RTCP。

12.万一是RTP，Offer运用特定负载编号来与某编码相对应，Answer应当维持这种对应牵涉,lo.longdishop.com。

13.Answer中的“m=”行编码应拥有优先次序，以便Offer能采纳最高优先级的选项。但即使如此，倡议Answer采纳与Offer相像的优先次序。

14.ptime指示领受的打包间隔，但并不请求两方的ptime值统一。但发送媒体流时应当遵照ptime指示的打包间隔来发送。

15.假使是RTP流，Answer应当采纳Offer提供的负载编码编号。

16.当Offer收到Answer后，必须采用Answer中的媒体类别中的一个，最终该当采纳排行的第一个。
